# 조언과 기대 성능

## 먼저 바로잡을 점

`google/gemma-4-31B-it-assistant`는 일반적인 instruction 모델 본체가 아니라 MTP drafter입니다. Hugging Face 모델 카드 기준으로 이 모델은 speculative decoding에서 메인 모델의 다음 토큰을 미리 제안하는 보조 모델로 설명되어 있습니다. 즉 일반 태스크 성능을 높이고 싶다면 보통 `google/gemma-4-31B-it`를 튜닝하고, `-assistant`는 추론 속도 개선에 씁니다.

## 어느 정도 성능을 기대할 수 있나

정확한 수치는 데이터 품질과 평가셋에 따라 크게 달라집니다. 아래는 실무적인 기대치입니다.

- 20~100 샘플: 말투, 응답 형식, 금지 문구 같은 스타일 정렬은 눈에 띄게 바뀔 수 있습니다.
- 500~2,000 샘플: 좁은 태스크에서 정답률, 응답 일관성, 포맷 준수율이 의미 있게 개선될 수 있습니다.
- 5,000~20,000 샘플: 특정 도메인에서 base 대비 확실한 차이를 만들 가능성이 큽니다.
- 50,000+ 고품질 샘플: 범용성 일부를 유지하면서 도메인 적응을 노려볼 수 있지만, 데이터 품질이 낮으면 오히려 역효과가 납니다.

Google Gemma 문서는 적은 경우 20개의 prompt-response pair만으로도 행동 변화를 줄 수 있다고 설명합니다. 다만 그건 "행동 변화 가능"에 가깝고, 리더보드 수준의 일반 성능 향상을 뜻하지는 않습니다.

## 리더보드 기준 기대치

- 고객지원/사내문서 QA 데이터만 넣으면 Open LLM Leaderboard 평균 점수는 유지 또는 하락 가능성이 큽니다.
- 한국어 업무형 태스크에서는 human eval 승률이나 task success rate가 크게 오를 수 있습니다.
- 범용 리더보드 상승을 원하면 한국어 업무 데이터만으로는 부족하고, 일반 추론과 코딩, 수학, 다국어 instruction 데이터가 함께 필요합니다.

## 권장 데이터 형식

가장 무난한 형식은 JSONL conversation 형식입니다.

```json
{"messages":[
  {"role":"system","content":"역할 정의"},
  {"role":"user","content":"질문"},
  {"role":"assistant","content":"정답"}
]}
```

권장 원칙:

- 한 샘플에는 한 가지 명확한 작업만 담습니다.
- `assistant` 답변은 실제 배포 때 원하는 답변 스타일과 최대한 같아야 합니다.
- 잘못된 답, 모호한 정책, 불필요하게 긴 추론 흔적은 넣지 않습니다.
- 훈련/검증 분리를 반드시 합니다.
- 내부 정책성 답변은 최신 기준으로만 만듭니다.

## 데이터 용량 추천

토큰 기준이 더 중요하지만, 대략적인 감으로 보면:

- PoC: 1MB~20MB
- 도메인 적응 시작점: 20MB~200MB
- 본격 학습: 200MB~수 GB

문장 수보다 중요한 것은 중복 제거와 정답 품질입니다. 같은 패턴의 문장을 많이 복제하는 것은 거의 도움이 안 됩니다.

## 학습 전략 추천

1. 1차는 `gemma-4-31B-it`에 QLoRA SFT를 겁니다.
2. 내부 평가셋에서 이득이 보이면 데이터 정제를 반복합니다.
3. 추론 배포 시 `gemma-4-31B-it-assistant`를 drafter로 붙여 속도를 높입니다.
4. 정말 리더보드를 노릴 때만 범용 데이터 혼합 또는 preference tuning을 추가합니다.
